## subject规则

subject是语义主体——这条信息是"关于谁/什么"的。

- 具体人名、公司名、理论名、角色名均可
  - "博采传媒成立于1993年" → subject: "博采传媒"
  - "Bowen家庭系统理论提出分化概念" → subject: "Bowen家庭系统理论"
  - "用户做过1周1500的项目" → subject: "用户"
- "谁传达的"不由subject承载，靠origin字段表达
- 禁止用描述性短语做subject（不是实体，是对一类信息的归纳）：
  - 错误: "用户识人需求性质"、"早期记忆记录"、"2024年末影视行业"、"定位精度影响因素"
  - 正确: "用户"、"AI"、"影视行业"、"影子定位法"
- 具体的、有名字的方法论/理论/原理/约束/概念都可以做subject
  - 判断标准：能用这个名字搜到或在对话中被反复用这个名字提及
  - 抽象原理、约束条件、共创结论只要有明确的名字，就是合格的subject（不要因为"抽象"就退回成描述性短语）
- **允许你自己为命名不明的概念浓缩出一个好名字**。如果原文清晰地描述了一个独立存在的方法/理论/原理/框架，但没给它起名（或名字冗长），你可以提炼一个简洁、准确、可复用的命名。
- **但必须严格区分"命名浓缩" vs "描述名词化"**，这是最容易犯错的地方：
  - **命名浓缩**（合格）：你为**一个独立存在、能被单独谈论的"东西"**起名。这个"东西"可以不依赖任何上下文被引用。
  - **描述名词化**（不合格）：你把"关于某事的一段描述"包了一层名词外壳，假装成一个实体。它无法脱离具体上下文被谈论。
- **判别口诀**：把候选 subject 放回一个新句子里问自己——**"它/它的做法是什么？"能独立答上来吗？**
  - "XX方法的做法是什么？" → 能答（指向一个独立方法） → ✅ 合格
  - "XX偏好是什么？" → 卡住，必须先问"谁的偏好、对什么的偏好" → ❌ 不合格，应该挂在"用户/某主体"下
  - "XX区别是什么？" → 卡住，必须先问"A 和 B 分别是什么" → ❌ 不合格，这是描述不是实体
- 归纳不出独立实体的信息，就老实挂在最大的相关主体上（"用户"、"AI"、具体人名/组织名）
- **"AI"就是"AI"，绝对不要写成"system"**。"system"只是origin_generator字段里的一个枚举值，表示"内容由AI产生"，它不是任何东西的名字。
  - AI分析了某件事 → subject: "AI"，origin_generator: "system"
  - AI推算出北纬30度 → subject: "AI"，predicate: "推算出"，origin_generator: "system"
  - AI建议用户暂缓离职 → subject: "AI"，predicate: "建议"，origin_generator: "system"
- subject必须是你能指着说"它做了/是了什么"的实体
